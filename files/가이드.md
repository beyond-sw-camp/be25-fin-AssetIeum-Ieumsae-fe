# 자산이음 프론트엔드 가이드

이 문서는 자산이음 프론트엔드 프로젝트를 처음 접한 작업자를 위한 안내서다.

프로젝트 설치 방법, 주요 기술, 실행 흐름, 폴더 구조, API 작성 방법과 개발 시 지켜야 할 기준을 설명한다.

## 1. 프로젝트 한눈에 보기

자산이음 프론트엔드는 Vue 기반의 단일 페이지 애플리케이션(SPA)이다.

```text
브라우저
  -> index.html
  -> src/main.ts
  -> Vue 애플리케이션
  -> Vue Router
  -> View 컴포넌트
  -> Pinia Store 또는 API 모듈
  -> Axios
  -> 백엔드 API
```

주요 기능 영역은 다음과 같다.

- 로그인과 사용자 권한
- 조직과 사원 관리
- 티켓 요청과 승인
- 유형자산과 무형자산 관리
- 구매와 예산 관리
- 전수조사
- 대시보드
- 알림과 로그

API와 요구사항의 상세 내용은 같은 폴더의 다음 문서를 참고한다.

- `API명세서.md`
- `요구사항 정의서.csv`

## 2. 주요 기술

| 기술 | 역할 |
|---|---|
| Vue 3 | 화면과 컴포넌트를 작성하는 프레임워크 |
| TypeScript | 데이터 형태를 검사하고 개발 중 오류를 줄이는 도구 |
| Vite | 개발 서버 실행과 운영 파일 빌드 |
| Vue Router | URL에 맞는 페이지 표시와 접근 권한 검사 |
| Pinia | 여러 화면에서 공유하는 상태 관리 |
| Axios | 백엔드 HTTP API 호출 |
| Tailwind CSS 4 | 클래스 조합을 이용한 화면 스타일 작성 |
| ESLint 10 | 코드 오류와 규칙 검사 |
| vue-tsc | Vue 파일을 포함한 TypeScript 검사 |

## 3. 필요한 실행 환경

이 프로젝트는 Node.js 24 이상을 기준으로 한다.

```text
Node.js >= 24
npm
```

Node 버전 기준은 다음 파일에 기록되어 있다.

- `.nvmrc`
- `package.json`의 `engines.node`

NVM을 사용한다면 프로젝트 폴더에서 다음 명령으로 버전을 맞출 수 있다.

```bash
nvm use
```

## 4. 처음 실행하는 방법

### 4.1 패키지 설치

```bash
npm install
```

팀원이 이미 확정한 `package-lock.json` 버전을 그대로 설치하려면 다음 명령을 권장한다.

```bash
npm ci
```

설치된 패키지는 `node_modules`에 저장된다. 이 폴더는 직접 수정하지 않는다.

### 4.2 환경변수 준비

`.env.example`을 참고하여 `.env.local`을 준비한다.

```dotenv
VITE_API_BASE_URL=http://localhost:8080/api/v1
VITE_SSE_BASE_URL=http://localhost:8080/api/v1
```

각 변수의 의미:

| 변수 | 의미 |
|---|---|
| `VITE_API_BASE_URL` | Axios가 요청할 백엔드 API 기본 주소 |
| `VITE_SSE_BASE_URL` | 실시간 알림을 구독할 서버 기본 주소 |

Vite에서 브라우저 코드가 읽어야 하는 환경변수는 반드시 `VITE_`로 시작해야 한다.

`VITE_` 환경변수는 최종 브라우저 코드에 노출될 수 있으므로 비밀번호, DB 정보, 비밀키를 넣으면 안 된다.

### 4.3 개발 서버 실행

```bash
npm run dev
```

Vite 개발 서버가 `http://localhost:5173`에서 실행되고 브라우저가 자동으로 열린다.

소스 파일을 수정하면 변경 내용이 빠르게 브라우저에 반영된다.

## 6. 애플리케이션 실행 흐름

### `index.html`

브라우저가 최초로 읽는 HTML 파일이다.

```html
<div id="app"></div>
<script type="module" src="/src/main.ts"></script>
```

`#app` 안에 Vue 화면이 들어가며 `src/main.ts`가 애플리케이션을 시작한다.

### `src/main.ts`

Vue 앱의 시작점이다.

주요 실행 순서:

1. Vue 앱 생성
2. Pinia 등록
3. 저장된 로그인 상태 복원
4. Vue Router 등록
5. 라우터 준비 대기
6. `#app`에 화면 표시

```ts
const app = createApp(App)
const pinia = createPinia()

app.use(pinia)

const auth = useAuthStore(pinia)
await auth.restoreSession()

app.use(router)
await router.isReady()

app.mount('#app')
```

### `src/App.vue`

애플리케이션의 최상위 컴포넌트다.

```vue
<template>
  <RouterView />
</template>
```

현재 URL에 맞는 화면을 표시하는 역할만 한다. 사이드바와 사용자 메뉴는 `AppLayout.vue`가 담당한다.

## 7. 폴더 구조

```text
src/
├─ api/          백엔드 API 요청 함수
├─ assets/       전역 CSS와 번들에 포함되는 자원
├─ components/   여러 화면에서 재사용하는 UI
├─ composables/  여러 컴포넌트에서 재사용하는 Vue 로직
├─ router/       URL, 페이지, 권한 연결
├─ stores/       Pinia 전역 상태
├─ types/        TypeScript 타입
├─ utils/        단순 변환과 표시용 유틸리티
├─ views/        URL별 페이지 컴포넌트
├─ App.vue       최상위 컴포넌트
└─ main.ts       애플리케이션 시작점
```

### `src/views`

라우터가 직접 표시하는 페이지를 작성한다.

예:

```text
views/auth/LoginView.vue
views/member/MemberListView.vue
views/ticket/TicketDetailView.vue
```

새 페이지를 만들었다면 `src/router/index.ts`에도 경로를 등록해야 한다.

### `src/components`

여러 화면에서 반복 사용하는 UI를 작성한다.

예:

- 버튼
- 테이블
- 모달
- 입력 폼
- 공통 레이아웃

특정 URL의 전체 페이지는 `views`, 페이지 안에서 재사용할 부분은 `components`에 둔다.

### `src/api`

백엔드 엔드포인트별 요청 함수를 작성한다.

```text
client.ts          Axios 공통 설정
auth.api.ts        인증 API
member.api.ts      사원 API
department.api.ts  부서 API
ticket.api.ts      티켓 API
asset.api.ts       자산 API
index.ts           API 모듈 모음
```

화면에서 Axios나 `fetch()`를 직접 호출하지 않고 이 폴더의 함수를 사용한다.

### `src/types`

API 요청, 응답과 화면 데이터의 TypeScript 타입을 작성한다.

```text
common.ts      공통 응답, 권한, 공통 상태
member.ts      로그인과 사원
department.ts  부서
ticket.ts      티켓
asset.ts       유형·무형 자산
budget.ts      예산과 대시보드 통계
survey.ts      전수조사
```

백엔드 응답 필드가 변경되면 API 함수뿐 아니라 관련 타입도 함께 확인한다.

### `src/stores`

여러 페이지가 공유해야 하는 상태를 Pinia로 관리한다.

현재 주요 Store:

- `auth.store.ts`: 사용자와 로그인 상태
- `department.store.ts`: 부서 목록과 선택 부서
- `notification.store.ts`: 화면 알림과 읽지 않은 알림 수

한 페이지에서만 사용하는 값은 Store보다 해당 컴포넌트의 `ref`나 `reactive`를 우선 사용한다.

### `src/composables`

Vue의 반응형 기능을 사용하는 재사용 로직을 작성한다.

현재 제공 기능:

- `useAsync`: 비동기 요청의 로딩과 오류 처리
- `usePagination`: 페이지 번호와 크기 관리
- `useFilter`: 검색 필터와 debounce
- `usePermission`: 역할별 권한 판단
- `useModal`, `useConfirm`: 모달 상태
- `useEventSource`: SSE 실시간 이벤트 구독

### `public`

가공 없이 그대로 배포할 파일을 둔다.

예:

- 파비콘
- 공용 SVG

코드에서 `/favicon.svg`처럼 루트 경로로 접근한다.

## 8. Vue 컴포넌트 기본 구조

이 프로젝트는 Vue의 `<script setup>` 문법을 사용한다.

```vue
<template>
  <button @click="increase">
    {{ count }}
  </button>
</template>

<script setup lang="ts">
import { ref } from 'vue'

const count = ref(0)

function increase() {
  count.value += 1
}
</script>
```

- `<template>`: 브라우저에 표시할 HTML 구조
- `<script setup lang="ts">`: 상태와 동작
- `ref`: 값이 바뀌면 화면도 다시 그려지는 반응형 값

템플릿에서는 `count`, TypeScript 코드에서는 `count.value`로 접근한다.

## 9. 라우터 사용법

라우터 설정은 `src/router/index.ts`에 있다.

기본 예:

```ts
{
  path: 'members',
  name: 'MemberList',
  component: () => import('@/views/member/MemberListView.vue'),
  meta: {
    title: '사원 목록',
    roles: ['SUPER_ADMIN'],
  },
}
```

- `path`: 브라우저 URL
- `name`: 코드에서 사용할 고유 이름
- `component`: 표시할 Vue 파일
- `meta.title`: 브라우저 제목
- `meta.roles`: 접근 가능한 역할

현재 역할 값:

| 역할 | 코드 |
|---|---|
| 최고 관리자 | `SUPER_ADMIN` |
| 부서책임자 | `DEPARTMENT_MANAGER` |
| 구매자산팀 | `ASSET_TEAM` |
| 사원 | `EMPLOYEE` |

프론트 라우터의 권한 검사는 화면 접근을 제어하기 위한 것이다. 실제 보안을 위해 백엔드도 반드시 권한을 검사해야 한다.

정적 경로는 동적 ID 경로보다 먼저 선언한다.

```ts
{ path: 'create', ... }
{ path: ':ticketId', ... }
```

순서가 반대이면 `/tickets/create`의 `create`가 티켓 ID로 인식될 수 있다.

## 10. Axios API 사용법

공통 Axios 설정은 `src/api/client.ts`에 있다.

```ts
export const httpClient = axios.create({
  baseURL: API_BASE_URL,
  timeout: 15_000,
})
```

공통 클라이언트가 처리하는 기능:

- 환경변수의 API 기본 주소 사용
- Access Token 자동 첨부
- 401 응답 시 토큰 갱신 시도
- 중복 토큰 갱신 요청 방지
- 공통 `ApiError` 생성
- JSON 요청
- 파일 업로드
- 파일 다운로드

### 공통 API 응답

백엔드 응답은 다음 형태를 기준으로 한다.

```ts
interface ApiResponse<T> {
  status: number
  errorCode: string | null
  message: string
  data: T
}
```

### GET 요청

```ts
import api from './client'
import type { Member, PageResponse } from '@/types'

export const memberApi = {
  getList: (params?: { page?: number; size?: number }) =>
    api.get<PageResponse<Member>>('/members', params),
}
```

사용:

```ts
const response = await memberApi.getList({ page: 0, size: 20 })
const members = response.data.content
```

### POST 요청

```ts
create: (body: MemberRegisterRequest) =>
  api.post<Member>('/members', body)
```

### PATCH 요청

```ts
changeDepartment: (memberId: number, body: DepartmentChangeRequest) =>
  api.patch(`/members/${memberId}/department`, body)
```

### 파일 업로드

```ts
uploadEvidence: (ticketId: number, file: File) => {
  const formData = new FormData()
  formData.append('file', file)

  return api.upload(`/tickets/${ticketId}/evidence`, formData)
}
```

`FormData`를 보낼 때 `Content-Type`을 직접 지정하지 않는다. 브라우저와 Axios가 boundary를 포함해 자동 설정한다.

### 파일 다운로드

```ts
const blob = await api.download('/reports/assets')
const url = URL.createObjectURL(blob)
```

사용이 끝나면 다음과 같이 URL을 해제한다.

```ts
URL.revokeObjectURL(url)
```

### 오류 처리

Axios 요청 오류는 `ApiError`로 전달된다.

```ts
import { ApiError } from '@/api/client'

try {
  await memberApi.getList()
} catch (error) {
  if (error instanceof ApiError) {
    console.error(error.status)
    console.error(error.errorCode)
    console.error(error.message)
  }
}
```

## 11. 로그인과 인증

로그인 상태는 `src/stores/auth.store.ts`가 관리한다.

로그인 성공 시 다음 값이 `localStorage`에 저장된다.

```text
accessToken
refreshToken
authUser
```

`src/main.ts`는 앱을 표시하기 전에 `restoreSession()`을 호출하여 저장된 로그인 정보를 복원한다.

Axios 요청 인터셉터는 Access Token이 있으면 자동으로 다음 헤더를 추가한다.

```http
Authorization: Bearer {accessToken}
```

백엔드가 401을 반환하면 Refresh Token으로 `/auth/refresh` 요청을 시도한다.

토큰 갱신 API의 실제 요청·응답 형식이 변경되면 `src/api/client.ts`의 `refreshAccessToken()`을 수정한다.

## 12. 실시간 알림

`src/composables/useEventSource.ts`는 SSE(Server-Sent Events) 기본 연결을 제공한다.

```ts
const {
  connect,
  on,
  disconnect,
  isConnected,
} = useEventSource()

connect()

const removeListener = on<Notification>('notification', (event) => {
  console.log(event.data)
})
```

컴포넌트가 제거되면 연결도 자동으로 종료된다.

브라우저 기본 `EventSource`는 임의의 Authorization 헤더를 넣을 수 없다. 현재 구현은 쿠키 인증을 전제로 `withCredentials: true`를 사용한다.

백엔드가 Bearer Token 방식으로 SSE 인증을 요구한다면 연결 방식을 백엔드와 함께 조정해야 한다.

@import "tailwindcss";

## 14. 주요 설정 파일

### `package.json`

프로젝트 이름, Node 버전, npm 명령어와 패키지 목록을 관리한다.

- `dependencies`: 브라우저에서 실제 사용하는 패키지
- `devDependencies`: 빌드, 타입 검사, 린트 등 개발 도구

### `package-lock.json`

설치할 패키지의 정확한 버전을 기록한다. 임의로 삭제하지 않는다.

패키지를 설치하거나 제거할 때 npm이 자동으로 갱신한다.

### `vite.config.ts`

- Vue 플러그인 등록
- `@` 경로 별칭 설정
- 개발 서버 포트 설정
- 브라우저 자동 실행 설정

`@`는 `src`를 의미한다.

```ts
import { useAuthStore } from '@/stores'
```

### `tsconfig.json`

TypeScript 설정의 시작점이다. 앱 설정과 Node 환경 설정을 연결한다.

### `tsconfig.app.json`

`src` 내부 TypeScript와 Vue 코드 검사 규칙을 관리한다.

주요 설정:

- 엄격한 타입 검사
- 사용하지 않는 변수와 매개변수 검사
- `@/*` 경로 별칭

### `tsconfig.node.json`

`vite.config.ts`처럼 Node.js에서 실행되는 설정 파일을 검사한다.

### `eslint.config.js`

ESLint 10 Flat Config 형식의 설정이다.

- `dist`
- `.env.local`을 포함한 `*.local`
- 로컬 설계 자료

## 15. 새 기능을 추가하는 순서

새 기능은 다음 순서로 작성하면 구조를 유지하기 쉽다.

1. 요구사항과 API 명세를 확인한다.
2. `src/types`에 요청과 응답 타입을 작성한다.
3. `src/api`에 백엔드 요청 함수를 작성한다.
4. 여러 화면이 공유하면 Store 또는 composable을 작성한다.
5. `src/views`와 `src/components`에 UI를 작성한다.
6. 새 페이지라면 라우터에 등록한다.
7. 타입 검사, 린트와 빌드를 실행한다.

```bash
npm run type-check
npm run lint
npm run build
```

## 16. 파일 배치 기준

다음 기준으로 파일 위치를 결정한다.

| 질문 | 위치 |
|---|---|
| URL 하나를 담당하는 전체 화면인가? | `views` |
| 여러 화면에서 반복되는 UI인가? | `components` |
| 백엔드 요청인가? | `api` |
| API 데이터 형태인가? | `types` |
| 여러 페이지가 공유하는 상태인가? | `stores` |
| Vue 반응형 공통 로직인가? | `composables` |
| 단순 문자열·날짜·라벨 변환인가? | `utils` |

## 17. 개발 시 주의사항

- 화면 컴포넌트에서 `fetch()`나 Axios를 직접 호출하지 않는다.
- API 요청은 `src/api/client.ts`와 도메인별 `*.api.ts`를 사용한다.
- API 응답 필드는 임의로 추측하지 않고 명세 또는 백엔드 구현을 확인한다.
- `any`보다 실제 타입을 작성한다.
- 토큰과 환경변수 이름을 파일마다 다르게 만들지 않는다.
- `node_modules`와 `dist`는 직접 수정하지 않는다.
- 기능 완료 전 `type-check`, `lint`, `build`를 실행한다.
- 프론트 권한 검사만 믿지 않고 백엔드 권한 검사를 함께 적용한다.
- 비밀정보를 `VITE_` 환경변수에 넣지 않는다.
